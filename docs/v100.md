# Nearr v100 Vision

Nearr is not just a social map app.

Nearr becomes a **decision engine for real-world action**.

The long-term product turns social intent into physical-world movement:

> See something online → save it → remember it at the right moment → decide to go → actually go.

Everything in this roadmap should reinforce four actions:

- discovering
- remembering
- deciding
- going

---

# Core Product Thesis

Nearr helps users collect real-world places from social media and turns those saved places into timely, useful decisions.

The long-term moat is not only the app UI. It is the system underneath it:

- extracting places from messy social content
- understanding user intent
- knowing what places matter to someone
- reminding them at the right time
- helping them decide what to do in real life

The highest-value version of Nearr is:

> **the place-intent layer for social media.**

---

# Layer 1: Core Engine

This has to be excellent before anything else matters.

## 1. Flawless Capture

Saving from social content must feel reliable.

Nearr should support:

- Instagram
- TikTok
- YouTube
- X/Twitter
- Reddit
- Pinterest
- generic web links

Nearr should understand:

- captions
- descriptions
- hashtags
- tagged handles
- profile bios
- video transcripts
- visible addresses
- restaurant/place names
- creator context

The goal is not just “parse a link.” The goal is:

> Given a social post, identify the real-world place the post is about.

Success target:

- Explicit restaurant name or address in the post should resolve correctly nearly every time.
- Generic posts should fail safely instead of guessing.
- Wrong auto-save should be treated as the worst outcome.

## 2. Social Evidence Layer

Nearr needs a consistent evidence layer for every platform.

This includes:

- post metadata
- creator bio
- tagged profile bios
- captions/descriptions
- transcripts
- links in bio
- visible addresses
- normalized place candidates

Near-term provider direction:

- Use Clipto for transcripts if API access and usage-based pricing work.
- Use HasData Instagram Profile API for Instagram bio/profile data.
- Keep providers behind interfaces so they can be swapped later.

Long-term provider direction:

- Use private/official API keys or approved API access from Instagram, TikTok, X, Reddit, Pinterest, YouTube, and other platforms where possible.
- Avoid relying permanently on third-party scraping APIs.
- Build in-house extraction only when it is cheaper, more reliable, or strategically necessary.

The long-term architecture should look like:

```text
Social link
→ Social metadata provider
→ Profile/bio provider
→ Transcript provider
→ Evidence normalizer
→ Place extraction model
→ Deterministic safety gate
→ Save / candidate confirmation / fallback
```

## 3. Address-First Resolution

If a post includes a real address, Nearr should not overthink.

Address or venue evidence should trigger deterministic resolution first:

- street address → Geocoding / Places verification
- venue + city → clean Places query
- tagged venue + city → clean Places query

Examples:

- “Crystal Fish in Monterey” → `Crystal Fish Monterey`
- “The C Restaurant in Monterey, CA” → `The C Restaurant Monterey CA`
- “Old Fisherman’s Grotto | Monterey CA” → `Old Fisherman’s Grotto Monterey CA`
- “39 Fishermans Wharf, Monterey, CA” → geocode / Places verify

Never use noisy social wrapper queries like:

- “Creator Name on Instagram…”
- “Travel influencer on TikTok…”
- “Food blogger on Instagram…”

## 4. Self-Hosted Model Layer

Nearr should eventually move away from depending on external AI APIs for core extraction.

Near-term:

- Use external APIs where they help ship faster.
- Compare Gemini, OpenAI, Anthropic, and xAI models on Nearr’s own link batches.
- Track speed, cost, JSON reliability, accuracy, and reasoning quality.

Long-term:

- Host Qwen, Llama, or another strong open-source model on Nearr-controlled infrastructure.
- Fine-tune or adapt the model around Nearr’s place-extraction task.
- Keep model outputs structured and testable.
- Use the model to propose places, but never let it bypass deterministic safety rules.

Why this matters:

- lower marginal cost at scale
- more control over latency
- more privacy/control over data
- ability to fine-tune on Nearr-specific examples
- less dependency on changing third-party model pricing or behavior

The model’s job:

> infer the likely real-world place from clean evidence.

The code’s job:

> decide whether that result is safe enough to save.

## 5. Deterministic Safety Gate

The model can propose. Code decides.

The safety gate controls:

- auto-save
- candidate confirmation
- manual fallback
- failed state

Rules:

- Wrong auto-save is the worst outcome.
- Correct candidate confirmation is better than manual fallback.
- Clean manual fallback is better than a wrong candidate.
- Raw handles alone should not auto-save.
- Generic food/travel text should not resolve to random places.
- Profile bio evidence is useful, but not automatic truth.
- Transcripts are evidence, but still need Places verification.

# Layer 2: Action-First Map

The map is not just a database of saved places. It should help users act.

## 1. Instant Map Action

Users should be able to:

- open a saved place on the map instantly
- see nearby saved places
- get directions quickly
- understand distance and radius visually
- avoid hunting through lists

Map behavior should feel simple:

- default to user location
- show saved-place bubbles clearly
- tap place → details / directions / reminder controls

## 2. Smart Reminders

This is one of Nearr’s strongest product hooks.

Examples:

- “You’re near X.”
- “You saved this 3 days ago.”
- “You still haven’t tried this.”
- “You’re near 3 places you saved from TikTok.”

The reminder system should be useful, not spammy.

Long-term reminders should account for:

- distance
- time of day
- user habits
- place category
- whether the user already visited
- how recently the place was saved

## 3. The “Go” Layer

Nearr should help users act, not just browse.

Actions:

- Open in Apple Maps / Google Maps
- Get directions
- Share with friend
- Add to plan
- Mark as visited
- Call / reserve / order if relevant

The product should make the jump from “saved” to “I’m going” feel natural.

# Layer 3: Decision Layer

This is where Nearr becomes part of daily life.

## 1. “What should I do right now?”

A strong Nearr session should answer:

> What should I do nearby?

Inputs:

- saved places
- current location
- distance
- time of day
- category
- past saves
- visited/unvisited status
- friends or creators followed

Outputs:

- places nearby
- best options right now
- quick filters
- simple decision UI

## 2. Multi-Category Expansion

Start with food because food is the clearest wedge.

Expand only after food works extremely well.

Future categories:

- food
- coffee
- activities
- events
- stores
- beaches
- hikes
- thrift shops
- date spots
- travel stops

The expansion should feel natural, not forced.

## 3. Revisit Memory

Nearr should bring back forgotten intent.

Examples:

- saved but not visited
- old saves nearby
- seasonal reminders
- “You saved this last month and you’re nearby now.”

This turns saved places into recurring value.

# Layer 4: Social Layer

Social comes after the core loop works.

## 1. Personal Maps

Users can curate:

- my saved spots
- my favorite restaurants
- date spots
- Santa Cruz places
- LA weekend list
- places to try

## 2. Follow People / Subscribe to Maps

Users can follow:

- friends
- creators
- local experts
- athletes
- food accounts
- travel accounts

This should feel like:

> Spotify playlists for real-world places.

Not Yelp reviews. Not generic feeds.

## 3. Trust-Based Discovery

Discovery should be based on people the user trusts.

Examples:

- “Your friend saved this.”
- “A creator you follow added this.”
- “People with similar saves also saved this.”
- “Three people you follow have this on their map.”

# Layer 5: Intelligence and Network Effects

This is the long-term moat.

## 1. Recommendation Engine

Recommendations should be based on:

- what users save
- what users visit
- what they ignore
- what similar users save
- what creators influence
- what is nearby
- what is timely

Examples:

- “People who saved this also saved…”
- “Trending near you this week.”
- “You might like this because you saved X.”

## 2. Intent Graph

The most valuable data is not only where people go.

It is:

> where people want to go.

Nearr can build a graph of:

- saved places
- planned visits
- creator influence
- social content that drives interest
- reminders that drive action
- saves that turn into visits

This is what makes Nearr strategically valuable.

# Layer 6: Experience Polish

This matters, but it should not distract from the core loop.

## 1. Clean, Sleek, UX-Focused UI

Nearr needs a full UI rework eventually.

The long-term interface should feel:

- clean
- fast
- sleek
- minimal
- map-first when useful
- action-first when deciding
- visually consistent
- easy for non-technical users

Design principles:

- fewer buttons
- clearer hierarchy
- stronger empty states
- better onboarding
- smoother animations
- more polished cards
- consistent dark/orange visual system
- no cluttered debug/product complexity in the main UI

Near-term UI goal:

> Make the product understandable and usable.

Long-term UI goal:

> Make Nearr feel like a premium consumer app.

## 2. Business Profiles

Later, Nearr can support richer place pages:

- photos
- hours
- reviews
- quick actions
- saved by friends
- creator videos
- notes
- visit history

Do not build this too early.

## 3. Widgets

Potential widgets:

- places near you now
- saved spots nearby
- weekend list
- reminders
- “try one of these today”

## 4. Gamification

Be careful.

Natural:

- places tried
- lists completed
- neighborhoods explored

Avoid:

- gimmicky streaks
- fake points
- anything that distracts from going places

# Things to De-Prioritize or Avoid

## 1. Premature Gamification

Gamification should not be used to cover up weak product value.

## 2. Social Before Utility

Do not build a social network before the app is useful alone.

## 3. Full UI Rework Too Early

The UI needs a major rework eventually, but the core product must work first.

Fix current usability issues now. Redesign later.

## 4. Overbuilding Extraction Before Measuring

Use third-party providers first when they save time.

Build in-house only when:

- cost is too high
- provider quality is poor
- provider latency is unacceptable
- data control becomes strategically important

## 5. Weak Auto-Save

Auto-save should only happen when Nearr is extremely confident.

Candidate confirmation is safer until the eval suite proves reliability.
