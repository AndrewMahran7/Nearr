# Nearr — iOS Share Extension

> Last updated: 2026-04-27
> Source of truth: Codebase (not assumptions)

> How the iOS "Save to Nearr" share-sheet target is wired today, what
> the current actual state is, and how to verify it end-to-end before
> shipping.

## TL;DR

- We use **`expo-share-extension` ^1.10.7** (declared in
  [package.json](../package.json) and registered as a plugin in
  [app.json](../app.json)). It generates the native target during
  `npx expo prebuild` so we don't hand-edit Xcode.
- The share-extension JS bundle entry is
  [index.share.js](../index.share.js) → [ShareExtension.tsx](../ShareExtension.tsx).
  Metro picks it up because [metro.config.js](../metro.config.js)
  registers `share.js` as a source extension.
- The active **App Group** is `group.com.nearr.ios` (verified in
  [ios/NearrShareExtension/NearrShareExtension.entitlements](../ios/NearrShareExtension/NearrShareExtension.entitlements)).
  Both the host app target and the extension target must be members of
  this group.
- The Supabase access token is published into that App Group's
  `UserDefaults` by the local Expo Module
  [modules/nearr-shared-auth](../modules/nearr-shared-auth/), via the
  `onAuthStateChange` listener in [lib/supabase.ts](../lib/supabase.ts).
- The extension calls the Supabase **Edge Function**
  [process-share-link](../supabase/functions/process-share-link/index.ts)
  with that token. Confident matches save silently; everything else
  hands off to the host app's `/share` route.
- The legacy [native/share-extension/](../native/share-extension/)
  scaffold uses the obsolete App Group `group.com.nearr.app` and is
  **dead code**. The placeholder plugin
  [plugins/withShareExtension.js](../plugins/withShareExtension.js) is
  a no-op — do NOT enable it.

## Runtime flow

```
┌─ user taps Share in Safari / Instagram / TikTok / Maps ─┐
│                                                          │
│ iOS shows share sheet → "Save to Nearr"                  │
│                                                          │
└──────────────────────────┬───────────────────────────────┘
                           ▼
            ShareExtension.tsx loads in extension process
                           │
                           ├── pickSharedUrl(props)
                           │     - props.url (Safari direct URL)
                           │     - first https? URL inside props.text
                           │       (Instagram / TikTok captions)
                           │
                           ├── EXPO_PUBLIC_PROCESS_SHARE_LINK_URL set?
                           │     no  → open_app
                           │     yes → continue
                           │
                           ├── sharedAuth.getToken() (App Group UserDefaults)
                           │     no  → open_app
                           │     yes → continue
                           │
                           ├── POST { url, accessToken }
                           │     Authorization: Bearer <token>
                           │     6-second timeout
                           │     any throw / non-200 → open_app
                           │
                           └── result.status:
                                 saved                → "Saved to Nearr ✓",
                                                        close()
                                 ambiguous            → handOffToHostApp(url)
                                 failed_requires_app  → handOffToHostApp(url)
                                 open_app             → handOffToHostApp(url)
                                                        (the realistic
                                                         default today)

handOffToHostApp(url):
   openHostApp("share?url=<encoded>")
   → expo-share-extension prepends scheme → nearr://share?url=…
   → host app boots, /share auto-runs the host-app pipeline
     documented in ARCHITECTURE.md §3 + §6
```

## App Group bridge

`group.com.nearr.ios` is the contract. Both targets must declare it
in their entitlements, and Apple Developer must list it on both App
IDs.

- Generated host-app entitlements (after `npx expo prebuild`):
  `ios/Nearr/Nearr.entitlements` should contain
  `<string>group.com.nearr.ios</string>` under
  `com.apple.security.application-groups`.
- Generated extension entitlements:
  [ios/NearrShareExtension/NearrShareExtension.entitlements](../ios/NearrShareExtension/NearrShareExtension.entitlements)
  contains `group.com.nearr.ios`.
- The Swift module reads/writes a single key in that App Group's
  `UserDefaults`. JS side:

  ```ts
  // modules/nearr-shared-auth/index.ts
  setToken(token: string | null): boolean
  getToken(): string | null
  clear(): boolean
  ```

  All three are defensive no-ops on Android / Expo Go / when the native
  module isn't autolinked.

- The host app calls `setToken` on **every** Supabase auth event (sign
  in, refresh, sign out → null) inside [lib/supabase.ts](../lib/supabase.ts).
  Refresh tokens are intentionally **never** shared — only the access
  token, so the worst-case stolen-token blast radius is one token
  lifetime.

## Apple Developer setup

Do this once per environment (dev team, prod team).

1. **Identifiers → App Groups** → add `group.com.nearr.ios`.
2. **Identifiers → App IDs**:
   - `com.nearr.ios` → enable App Groups capability → check
     `group.com.nearr.ios`.
   - `com.nearr.ios.NearrShareExtension` → enable App Groups capability
     → check `group.com.nearr.ios`.
3. Regenerate provisioning profiles for both App IDs.
4. EAS will pick up the new profiles on the next build (or run
   `eas credentials` to force a refresh).

## Build + run

```sh
npx expo prebuild --clean
eas build --profile development --platform ios
# install on a real device — the share extension does NOT show up in
# Simulator share sheets reliably.
```

After install, sign in to the host app at least once so the JWT lands
in the App Group `UserDefaults`. Then:

- Open Safari → share an `instagram.com` / `tiktok.com` / `google.com/maps`
  URL → tap "Save to Nearr".
- Watch logs via `npx expo run:ios --device` or Xcode's Console with
  filter `Nearr`.

## Verification checklist

Run all of this on a real device with an EAS dev build before claiming
the share extension works:

- [ ] `ios/NearrShareExtension/NearrShareExtension.entitlements`
      contains `group.com.nearr.ios`.
- [ ] Host-app generated entitlements contain `group.com.nearr.ios`.
- [ ] Apple Developer App IDs both list the App Group.
- [ ] `EXPO_PUBLIC_PROCESS_SHARE_LINK_URL` is set in the build's env
      and points at a deployed function (`curl -X OPTIONS` returns
      CORS headers).
- [ ] `supabase secrets list` shows both `GEMINI_API_KEY` (optional
      but recommended) and `GOOGLE_PLACES_KEY`.
- [ ] After signing in to the host app and killing it, sharing a
      known-good Instagram reel URL produces "Saved to Nearr ✓"
      within ~6 s, and the row appears in Supabase
      `saved_places`.
- [ ] After signing out, the same share gracefully hands off to the
      host app's `/share` (no silent save with stale token).
- [ ] Sharing a URL whose page returns 404 produces a host-app
      handoff (no crash, no silent failure).
- [ ] Killing the function (rename it temporarily) produces a
      host-app handoff (no crash, no toast).

## Known limitations

- The extension has no UI for ambiguous results — it always defers to
  the host app for those.
- 6-second timeout is conservative; some Instagram pages take longer
  to OG-resolve. Tune in [ShareExtension.tsx](../ShareExtension.tsx)
  if you see false `open_app` outcomes.
- iOS does not always show the extension in Simulator share sheets.
  Test on hardware.
- The extension shares only the **access token**, never the refresh
  token. If the access token is expired and the extension fires before
  the host app refreshes it, the Edge Function's `auth.getUser` call
  will fail and the extension will hand off to the host app — this is
  intentional, not a bug.
